How many people think they could - on the first try - code an XHR request with a callback with no syntax errors?

Without Googling, try to code an XHR that gets a JSON file with a callback.

My first attempt

var xhr = new XMLHTTPRequest();
xhr.onload = function( data ) {
    console.log( data );
};
xhr.open( "test.json" );
xhr.send();
  • XMLHttpRequest (capitalization)
  • data --> this.responseText
  • open() takes "get"

Correct

var xhr = new XMLHttpRequest();
xhr.onload = function() {
    console.log( this.responseText );
};
xhr.open( "get", "test.json" );
xhr.send();
How many people can retrieve this JSON file using jQuery?

jQuery

$.getJSON( "test.json" ).then(function( data ) {
    console.log( data );
});
Less verbose APIs are easier to remember, easier to explain to beginners.

    return Array.prototype.slice.call(
        document.querySelectorAll( selector )

Importance of reducing round trips

  • In 2012, the average mobile RTT (Round Trip Time) time was 344ms in the US and 372ms in the UK.
  • In 2012 the average mobile download throughput was 1.6 Mbps in the US and 1.4 Mbps in the UK.
(source)

Reducing round trips > Reducing file size

So what does this mean for jQuery?

Yahoo's Performance Best Practices


http://developer.yahoo.com/performance/rules.html

Concatenate jQuery with your app's other scripts

<!-- before -->
<script src="jquery.js"></script>
<script src="app.js"></script>

<!-- after -->
<script src="app.js"></script>
If you're using jQuery then you have another JavaScript file for the things that your app does.

Don't load jQuery from a 3rd party CDN.

<script src="//ajax.googleapis.com/ajax/libs/jquery/2.1.0/jquery.min.js"></script>

Main anti-jQuery arguments:

  1. API Parity
    Native APIs are verbose, not easy to use, and there are lots of gotchas.
  2. Too Big / Bloated
    Make sure you concatenate, minify, cache, and gzip your scripts.
  3. Performance
    Don't prematurely optimize. If you detect delays in your UI, optimize then.

jQuery isn't for everyone

  • If you have a single button with a click handler, don't use jQuery.
  • If you're writing a library, consider whether you really need jQuery as a dependency (just as you would any other library).

jQuery's APIs are easier to

  • write
  • maintain
  • refactor
  • debug
  • test
  • grep
  • Google
  • teach
  • understand
  • extend
But it's important to remember that in the software development industry we're paid to ship software, and if jQuery makes it easier for you to do that, then use it; and don't feel guilty about it.
